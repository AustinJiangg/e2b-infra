# 06 日常运维手册

本篇是部署完成后的**日常操作手册**：每个组件怎么看状态、看日志、重启，Nomad/Consul 怎么用，
以及一页纸巡检清单。改动类操作（改代码/改配置怎么生效）不在本篇——那是
`single-node-offline-deploy.md` §5 场景表的职责。

## 1. 开工第一步：拿到 token

几乎所有 Nomad 操作都需要 ACL token。标准起手式：

```bash
cd /opt/e2b-infra
source .env                          # 读入 NOMAD_ACL_TOKEN / CONSUL_ACL_TOKEN
export NOMAD_TOKEN=$NOMAD_ACL_TOKEN  # ⚠️ 必须在 source 之后：.env 里有一行 `export NOMAD_TOKEN=`
                                     #    （空值）会把你先设的值冲掉
export CONSUL_HTTP_TOKEN=$CONSUL_ACL_TOKEN
```

设好 `NOMAD_TOKEN` 环境变量后，后面所有 nomad 命令**不用再带 `-token`**。

**token 异常自检**（命令 403 时先跑这个）：

```bash
grep NOMAD_ACL_TOKEN /opt/e2b-infra/.env        # 是真 UUID 还是占位符？
cat /data/nomad/acl.token; echo                  # 集群侧的真相（600 权限，需 root）
# 两者不一致 → 把 acl.token 的值写回 .env（或按 runbook §6.1 处理）
```

## 2. Nomad 使用教程

### 2.1 概念 30 秒

- **job**：一份部署声明（来自 `rendered/*.hcl`）。本部署 4 个：`api`、`client-proxy`、
  `redis`、`template-manager-system`；
- **allocation（alloc）**：job 的一次落地实例（job 的某个 task group 在某节点上的运行体）。
  看日志、进容器都以 alloc 为对象；
- **evaluation（eval）**：一次调度决策记录，排查"为什么没调度起来"时看；
- **node**：客户端节点（本部署 1 个，server+client 合体）；**node pool**：节点分组
  （本部署都在 `api` 池）。

### 2.2 状态查询

```bash
nomad job status                          # 所有 job 一览（Status=running 才正常）
nomad job status template-manager-system  # 单个 job：版本、alloc 列表、部署健康
nomad alloc status <alloc_id>             # 单个 alloc：任务事件、资源、端口（id 打前几位即可）
nomad node status                         # 节点列表（ready 才正常）
nomad node status -self -verbose          # 本节点详情（属性、驱动健康、资源占用）
nomad node pool list                      # 节点池（api / build / default / all）
nomad job inspect api                     # job 的完整 JSON spec（查 env 是否生效必备）
nomad eval list -job api                  # 调度记录（pending 排查入口）
nomad server members                      # server 成员（alive + leader 标记）
curl -s http://127.0.0.1:4646/v1/status/leader   # 有 leader 即 server 正常（此接口免 token）
```

### 2.3 看日志（最常用）

```bash
# 方式一：按 job 直达（随机挑一个 running alloc；单机就一个，正好）
nomad alloc logs -job template-manager-system        # stdout
nomad alloc logs -stderr -job template-manager-system
nomad alloc logs -f -job api                         # -f 跟随（类 tail -f）

# 方式二：精确到 alloc（多 alloc 或看历史时）
nomad job status template-manager-system             # 先抄 alloc id
nomad alloc logs <alloc_id>                          # 任务名唯一时可省略任务名
nomad alloc logs <alloc_id> start                    # 多任务时指定任务名（如 api/template-manager 的
                                                     # 主任务叫 start；以 nomad alloc status 显示为准）

# orchestrator 的沙箱日志（zap JSON）常用过滤：
nomad alloc logs -job template-manager-system | grep '\[ResumeSandbox\]' | tail -20
nomad alloc logs -job template-manager-system | jq -r 'select(.level=="error")' 2>/dev/null | tail
```

注意：

- **orchestrator 的日志在 `template-manager-system` job 里**（没有单独的 orchestrator job，
  01 篇 §3.3 的误区一）；
- alloc 重启会换 id，旧 stdout 随之难找——压测期间别重启该 job（benchmark 的
  `collect_logs.sh` 会遍历全部 running alloc 收齐日志）；
- 日志文件实体在 `/data/nomad/alloc/<alloc_id>/alloc/logs/`，nomad 命令不可用时可直接去翻。

### 2.4 Web UI

浏览器打开 `http://SERVER_IP:4646/ui` → 右上角 **ACL Tokens** 里粘贴 `NOMAD_ACL_TOKEN` →
Jobs 页可视化看 job/alloc/日志/资源曲线。Consul UI 同理：`http://SERVER_IP:8500/ui`
（token 用 `CONSUL_ACL_TOKEN`）。

### 2.5 常用操作

```bash
# 重启一个 job（换二进制后强制重建 alloc；hcl 没变时 run 是 no-op，必须先 stop）
nomad job stop template-manager-system
nomad job run rendered/template-manager.hcl          # 提交幸存的渲染产物（env 不变）

# 只改了 env → 用 build.sh 的快捷方式（render + run，env 有 diff 时 nomad 自动换 alloc）
bash /opt/e2b-infra/build.sh -r template-manager

# 全量重部署（改了 .env 之后）
bash /opt/e2b-infra/build.sh -f

# 优雅停/删 job
nomad job stop api                                   # 停（保留历史）
nomad job stop -purge api                            # 停并清除记录

# 垃圾回收（清理 dead alloc / 完成的 eval）
nomad system gc

# 进入容器化任务调试（docker driver 的 job 可用）
nomad alloc exec -task start <api_alloc_id> /bin/sh
```

> ⚠️ `nomad job stop template-manager-system` 会**杀掉该节点上所有运行中的沙箱**，挑好时机。

### 2.6 排查"job 起不来"的套路

```bash
nomad job status <job>            # 看 Placement Failures / alloc 状态
nomad alloc status <alloc>        # 看 Recent Events：镜像拉不下来？端口占用？OOM？
nomad alloc logs -stderr <alloc>  # 进程自己报什么错
nomad eval list -job <job>        # blocked eval = 资源/约束不满足
journalctl -u nomad -e            # agent 层面的错误（driver、指纹、consul 连接）
```

典型原因：Harbor 没起（拉镜像失败）→ 先修 §5.2；端口被占 → `ss -tlnp | grep <port>`；
raw_exec 任务二进制缺失 → `/usr/bin/template-manager` 是否存在/可执行。

## 3. Consul 使用教程

```bash
consul members                                    # 集群成员（alive 才正常）
consul catalog services                           # 已注册服务：api、client-proxy、redis、template-manager…
consul catalog nodes                              # 节点
dig @127.0.0.1 -p 8600 redis.service.consul       # 服务 DNS 直查
dig redis.service.consul                          # 走 dnsmasq 转发链路（验证 §6 DNS 链）
curl -s -H "X-Consul-Token: $CONSUL_HTTP_TOKEN" \
     http://127.0.0.1:8500/v1/health/service/template-manager | jq '.[].Checks[].Status'
                                                  # 健康检查状态（passing 才正常）
```

服务是 Nomad 按 job 里的 `service {}` 块自动注册的；`REDIS_URL=redis.service.consul`
这类地址能通全靠"dnsmasq → Consul DNS"这条链。

## 4. systemd 组件（nomad/consul/minio/nginx/dnsmasq/docker）

通用三板斧：

```bash
systemctl status <svc> --no-pager     # 状态 + 最近日志
journalctl -u <svc> -e                # 完整日志（-f 跟随，--since "10 min ago" 圈时间）
systemctl restart <svc>               # 重启
```

各组件补充：

| 服务 | 重启影响 | 专属检查 |
|---|---|---|
| `nomad` | client 重新做网络指纹，**4646 可能几十秒~几分钟不可用**（veth 多时，runbook §6.5）；job/沙箱进程本身不死 | `ss -tlnp | grep -E ':4646|:4647'` |
| `consul` | 短暂服务发现抖动；nomad 会自动重连 | `consul members` |
| `minio` | 期间模板读写失败；重启秒级 | `curl -s http://$SERVER_IP:9000/minio/health/ready -o /dev/null -w '%{http_code}\n'` |
| `nginx` | 期间模板构建拉 harbor:443 失败 | `nginx -t`（改配置后先校验再 reload） |
| `dnsmasq` | 期间 *.e2b.app / *.consul 解析失败 | `dig test.e2b.app` 应回 127.0.0.1 |
| `docker` | ⚠️ **重量级**：postgres/harbor/所有 docker-driver 的 nomad 任务全部重启 | 挑没有沙箱在跑的时机 |

## 5. Docker 层组件

### 5.1 PostgreSQL

```bash
docker ps -f name=postgres                        # 应 healthy
docker logs --tail 50 postgres
docker restart postgres
docker exec -it postgres psql -U postgres -d mydatabase    # 进库
```

常用 SQL（库名 `mydatabase`，密码 `local`）：

```sql
\dt                                               -- 看有哪些表（迁移是否跑过）
SELECT * FROM teams;                              -- 初始团队 E2B 在不在
SELECT id, concurrent_instances, max_length_hours FROM tiers;   -- 配额（deploy.sh 放开到 10000）
```

备份/恢复（数据在容器可写层，**没挂卷**，rm 容器=丢库）：

```bash
docker exec postgres pg_dump -U postgres mydatabase > /root/backup-$(date +%F).sql
cat backup.sql | docker exec -i postgres psql -U postgres -d mydatabase
```

### 5.2 Harbor

```bash
cd /opt/e2b-infra/harbor
docker-compose ps                                 # 全组容器应 Up（healthy）
docker-compose logs --tail 50 core registry proxy
docker-compose restart                            # 整组重启
# 起不来时的完整重装（配置不变）：docker-compose down && bash install.sh

# 仓库内容检查（registry API）
curl -s -u admin:Harbor12345 http://$SERVER_IP:2900/v2/_catalog | jq
curl -s -u admin:Harbor12345 http://$SERVER_IP:2900/v2/e2b-orchestration/api/tags/list | jq
```

UI：`http://SERVER_IP:2900`（admin/Harbor12345），项目 `e2b-orchestration`。
遇到 registry/proxy 容器反复重启，优先怀疑配置属主问题：
`chown -R 10000:10000 common/config/{registry,nginx} && docker-compose restart registry proxy`
（05 篇 §2 的已知坑）。

### 5.3 redis（在 Nomad 里，不是 systemd）

```bash
nomad job status redis
docker ps -f name=redis                           # nomad docker driver 起的容器
docker exec -it $(docker ps -qf name=redis) redis-cli ping     # PONG
docker exec -it $(docker ps -qf name=redis) redis-cli dbsize   # 沙箱状态键数量
```

## 7. E2B 业务层自检

```bash
cat /root/.e2b/config.json                         # SDK 凭据（teamId/accessToken/teamApiKey）
# 最小可用性验证（api → orchestrator → FC → envd 全链路）：
cd /path/to/benchmark && bash sync-env.sh          # 同步凭据到 benchmark/.env
python run_benchmark.py --template base --count 1 --warmup 0 --kill-each

# 进程侧确认
pgrep -af '/usr/bin/template-manager'              # orchestrator 合体进程
pgrep -c firecracker                               # 当前沙箱（VM）个数
ps -o pid,lstart,cmd -p $(pgrep -f /usr/bin/template-manager | head -1)   # 进程起始时间
tr '\0' '\n' < /proc/$(pgrep -f /usr/bin/template-manager | head -1)/environ \
  | grep -E 'E2B_FC_LAUNCH_MODE|MAX_STARTING|ORCHESTRATOR_SERVICES'       # 运行 env 终裁
```

沙箱列表/清理用 SDK（`Sandbox.list()` / `.kill()`，示例见 `docs/zh/maintain.md`）。

## 8. 宿主机专项检查

### 8.1 沙箱基础设施

```bash
lsmod | grep nbd && ls /dev/nbd* | wc -l          # 自编译 nbd 模块在位、512 个设备
grep Huge /proc/meminfo                            # HugePages_Total/Free、nr_overcommit
mount | grep -E 'snapshot-cache|hugepages'         # 各恰好 1 条（叠加了按 runbook §7 清）
ls /fc-versions/v1.12.1/firecracker /fc-kernels/vmlinux-6.1.102/vmlinux.bin /fc-envd/envd
ip netns list | head                               # 沙箱网络命名空间
iptables -t nat -L PREROUTING -n | grep 3002       # 80→3002 规则在位
```

### 8.2 资源水位

```bash
free -h; swapon --show
df -h / /root/data/minio /data /orchestrator /mnt/snapshot-cache
ip -o link | grep -c veth                          # 邻居容器 veth 数（>1000 警惕 nomad 重启变慢）
```

### 8.3 DNS 链路

```bash
dig test.e2b.app +short                            # → 127.0.0.1
dig redis.service.consul +short                    # → 本机 IP（经 dnsmasq→consul:8600）
grep -c '^nameserver 127.0.0.1' /etc/resolv.conf   # 恰好 1
```

### 8.4 服务器重启后的恢复动作（顺序执行）

机器重启后**不会全自动恢复**，按此清单补：

```bash
# 1) 手动装自编译 nbd（每次重启必做，runbook §0.2）
rmmod nbd 2>/dev/null || true
insmod /path/to/nbd.ko nbds_max=512

# 2) SELinux 回 permissive（-i 里的 setenforce 不持久）
setenforce 0

# 3) 核对应当在跑的服务（docker/postgres/harbor 靠 restart 策略自恢复；
#    nomad/consul/minio/nginx 已 enable；dnsmasq 视安装方式可能没 enable，起不来就 enable --now）
systemctl status docker nomad consul minio nginx dnsmasq --no-pager | grep -E 'service|Active'

# 4) 挂载与大页由 start-client/init-client 布置，重启后丢失 → 重跑幂等脚本
bash /opt/e2b-infra/start-client.sh api $SERVER_IP && bash /opt/e2b-infra/init-client.sh

# 5) iptables 规则重启后丢失 → 重加（05 篇 §11 那两条），或重跑 -s 的等价段
# 6) nomad job 状态检查，必要时 build.sh -f
```

## 9. 变更操作速查（指路）

| 想干什么 | 用什么 |
|---|---|
| 改了 Go 源码 | runbook §5 场景二（重建 RPM → 重放 overlay → cp /usr/bin → stop+run） |
| 改单个 job 的 env | 场景三（改 `nomad/<job>.hcl` → `build.sh -r <job>` → 四层验证） |
| 改 `.env` 部署变量 | 场景四（`build.sh -f`） |
| 改 nomad/consul 自身配置 | 场景五（改 `/etc/nomad.d/default.hcl` → `systemctl restart nomad`） |
| 彻底重来 | 场景一（`build.sh -d` → `-i` → `-s`） |
| 制作沙箱基础镜像 | `build.sh -m <镜像>`（往镜像里灌 systemd/ssh/websocat 等，推成 `.../openclaw-openviking:custom`） |
| 压测/瓶颈定位 | `benchmark/README.md` + `高并发瓶颈定位方案.md` |
| 403 / 端口等待 / 脏数据 | runbook §6（6.1/6.5/§7 对号入座） |

## 10. 一页纸巡检脚本

```bash
#!/bin/bash
# e2b 单机健康巡检（只读，可随时跑）
source /opt/e2b-infra/.env; export NOMAD_TOKEN=$NOMAD_ACL_TOKEN
echo "== nomad ==";   curl -s http://127.0.0.1:4646/v1/status/leader; echo
nomad job status | awk 'NR==1 || /running|pending|dead/'
echo "== consul ==";  consul members 2>/dev/null | head -3
echo "== 依赖 ==";
docker ps --format '{{.Names}}\t{{.Status}}' | grep -E 'postgres|harbor' | head -5
systemctl is-active minio nginx dnsmasq docker consul nomad | paste -sd' '
curl -s -o /dev/null -w 'minio:%{http_code} ' http://$SERVER_IP:9000/minio/health/ready
curl -s -o /dev/null -w 'harbor:%{http_code}\n' http://$SERVER_IP:2900/api/v2.0/ping
echo "== 沙箱底座 ==";
echo "nbd:$(ls /dev/nbd* 2>/dev/null | wc -l) veth:$(ip -o link | grep -c veth) fc:$(pgrep -c firecracker)"
grep -E 'HugePages_(Total|Free)' /proc/meminfo | paste -sd' '
mount | grep -cE 'snapshot-cache|hugepages' | xargs echo "特殊挂载条数(应为2):"
iptables -t nat -C PREROUTING -p tcp --dport 80 -j REDIRECT --to-port 3002 2>/dev/null \
  && echo "iptables 80→3002: OK" || echo "iptables 80→3002: 缺失!"
echo "== 磁盘 ==";    df -h / /root/data/minio 2>/dev/null | awk 'NR==1||/dev/'
```

跑完任一项异常，回到对应小节处置；处置不了的翻 runbook §6。
