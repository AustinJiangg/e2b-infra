# 01 整体架构与组件总览

本篇回答一个问题：**执行完 `rpm -ivh` + `build.sh -i` + `build.sh -s` 之后，这台服务器上到底跑着什么，它们怎么协作。**

## 1. 一句话总览

E2B 是一套"AI 代码沙箱"基础设施：每个沙箱是一台 **Firecracker microVM**。
本仓库把上游（面向 GCP 多节点集群）的 e2b-infra 改造成 **单台 ARM 服务器、离线可装** 的形态：

- 用 **RPM** 把 Go 二进制、部署脚本、内核镜像等打成一个包装到 `/opt/e2b-infra`；
- 用 **`build.sh -i`** 装齐周边依赖（PostgreSQL、MinIO、Harbor、Nginx、E2B SDK）；
- 用 **`build.sh -s`** 拉起控制面（Consul + Nomad）并把 E2B 的业务组件作为 Nomad job 部署上去。

## 3. 组件清单（运行形态 / 端口 / 数据 / 日志）

### 3.1 控制面与宿主服务（systemd 管理）

| 组件 | 运行形态 | 端口 | 数据/配置位置 | 日志 |
|---|---|---|---|---|
| **Nomad** | systemd `nomad.service`，单 agent 同时开 server+client | 4646 HTTP/UI，4647 RPC，4648 serf | 配置 `/etc/nomad.d/default.hcl`，数据 `/data/nomad`（含 ACL token 持久化 `/data/nomad/acl.token`） | `journalctl -u nomad` |
| **Consul** | systemd `consul.service` | 8500 HTTP/UI，8600 DNS，8300-8302 集群 | 数据 `/opt/consul`（含 `/opt/consul/acl.token`） | `journalctl -u consul` |
| **MinIO** | systemd `minio.service` | 9000 S3 API，9001 控制台 | 数据 `/root/data/minio`，配置 `/etc/default/minio.yml` | `journalctl -u minio` |
| **Nginx** | systemd `nginx.service` | 443（https 反代 Harbor），80（被 iptables 遮蔽） | `/etc/nginx/nginx.conf`，证书 `/etc/nginx/ssl/harbor.{crt,key}` | `/var/log/nginx/` |
| **dnsmasq** | systemd `dnsmasq.service` | 53 DNS | `/etc/dnsmasq.conf` + `/etc/dnsmasq.d/consul.conf` | `journalctl -u dnsmasq` |
| **Docker** | systemd `docker.service`（**自备，不由脚本安装**） | - | `/etc/docker/daemon.json`（需含 insecure-registries `SERVER_IP:2900`） | `journalctl -u docker` |

### 3.2 Docker 直跑的依赖（不归 Nomad 管）

| 组件 | 运行形态 | 端口 | 说明 |
|---|---|---|---|
| **PostgreSQL** | `docker run --name postgres --restart=always` | 5432 | 元数据库 `mydatabase`（teams/tiers/模板等），密码 `local`；`build.sh -i` 拉起 |
| **Harbor** | `/opt/e2b-infra/harbor/` 下 docker-compose 一组容器 | 2900 HTTP | 私有镜像仓库，项目 `e2b-orchestration`，账号 `admin/Harbor12345`；`-i` 解压配置、`-s` 才真正安装启动 |

### 3.3 Nomad job（业务面，`deploy.sh` 提交）

| Nomad job 名 | 来源 hcl | driver | 任务内容 | 端口 |
|---|---|---|---|---|
| `api` | `nomad/api.hcl` | docker | 主任务 `api` 镜像（REST API）；**prestart 任务 db-migrator** 先跑数据库迁移 | 3000 REST，5009 gRPC |
| `client-proxy` | `nomad/edge.hcl` | docker | client-proxy 镜像（又称 edge）：沙箱流量代理 + 边缘 API | 3002 proxy，3001 edge-api，3003 health |
| `redis` | `nomad/redis.hcl` | docker | redis:7.4.4-alpine（从 Harbor 拉），沙箱运行态存储（`SANDBOX_STORAGE_BACKEND=redis`） | 6379 |
| `template-manager-system` | `nomad/template-manager.hcl` | **raw_exec** | 直接跑宿主机二进制 `/usr/bin/template-manager --port 5008`；`ORCHESTRATOR_SERVICES="orchestrator,template-manager"` 让**一个进程同时承载 orchestrator（跑沙箱）和 template-manager（构建模板）两个服务** | 5008 gRPC |

> ⚠️ 两个容易误解的点：
> 1. **没有单独的 orchestrator job**。上游有 `orchestrator.hcl`，但本部署里 orchestrator 逻辑合并进
>    `template-manager-system` 这个 job（见上表）。`pgrep -f orchestrator` 找不到进程是正常的，
>    要按 `template-manager` 找。
> 2. `edge.hcl` 里定义的 job 名叫 **`client-proxy`**，文件名和 job 名不一致。

### 3.4 沙箱层（每个沙箱一组）

| 组件 | 说明 |
|---|---|
| **firecracker** | microVM 监视器进程，二进制 `/fc-versions/v1.13.1/firecracker`（RPM 自带 ARM 定制版，基于 v1.12.1；目录名是代码查找的版本标签）。每个沙箱一个进程，跑在独立 netns 里 |
| **vmlinux** | 客户机内核 `/fc-kernels/vmlinux-6.1.102/vmlinux.bin`（RPM 自带 ARM 版，另有 openEuler 变体） |
| **envd** | 沙箱 VM **内部**的代理进程（`/fc-envd/envd` 在模板构建时注入 rootfs），SDK 的命令执行/文件操作最终都由它执行 |
| **rootfs（nbd）** | 沙箱根盘通过 NBD 设备提供，依赖自编译 `nbd.ko`（`nbds_max=512`，已一次性固化、**开机自动加载**，固化步骤见 `single-node-offline-deploy.md` §0.2） |
| **内存（uffd + hugepages）** | 快照恢复用 userfaultfd 按需加载页，大页在 `init-client.sh` 里预留 |

## 4. 三条关键链路

### 4.1 创建沙箱（最常用路径）

```
SDK Sandbox.create("base")
  → http://SERVER_IP:3000  (api job；鉴权查 postgres 的 team/api key)
  → gRPC 5008              (template-manager-system 进程里的 orchestrator 服务)
  → 准入排队（MAX_STARTING_INSTANCES_PER_NODE 信号量）
  → 从模板快照恢复：连 nbd rootfs ∥ 取网络槽位(netns) ∥ 读模板元数据
  → 拉起 firecracker 进程（E2B_FC_LAUNCH_MODE 三档优化，见 benchmark/ 两篇 FC 启动优化文档）
  → load snapshot + resume VM（uffd 供页）
  → 同步调 VM 内 envd /init → 就绪
  → 沙箱状态写 redis
```

各阶段耗时埋点（`[ResumeSandbox]` 日志）与压测方法见 `benchmark/README.md`。

### 4.2 访问沙箱（数据面）

```
SDK/浏览器访问 http://<sandboxid>.e2b.app:80
  → dnsmasq: *.e2b.app → 127.0.0.1（本机部署时在部署机上解析）
  → iptables nat: 80 → 3002（PREROUTING 管外部流量，OUTPUT -o lo 管本机流量）
  → client-proxy(:3002) 按 Host 头找到对应沙箱 → 转发进 VM（envd/用户进程端口）
```

### 4.3 构建模板（`Template.build`）

```
SDK Template.build(from_dockerfile('FROM harbor:443/e2b-orchestration/ubuntu:22.04-custom'))
  → api → template-manager 服务（同一进程）
  → 从 Harbor 拉基础镜像：走 nginx 的 https://harbor:443 反代（E2B 构建器只认 https，
    自签证书已装入系统信任库；`harbor` 主机名需能解析到本机——脚本不自动写 /etc/hosts，
    解析不到时手动加一行 `127.0.0.1 harbor`）
  → 展开镜像 → 注入 busybox systeminit + envd + provision.sh 做成 rootfs
  → 起一次 VM 跑构建 → 产出模板（rootfs + 内存快照 + 元数据）→ 存 MinIO 桶
  → 之后创建沙箱都从该快照恢复（所以叫 ResumeSandbox）
```

## 5. 配置与凭据从哪来（新人最容易懵的部分）

| 文件 | 角色 |
|---|---|
| `/opt/e2b-infra/dep/.env` | **部署基线**（随 RPM 铺下来，来自仓库 `e2b-deploy/dep/.env`）。`build.sh` 启动时 source 它拿 `SERVER_IP`；里面的 ACL token 是**占位符** |
| `/opt/e2b-infra/.env` | **运行时真相**。`build.sh -i` 用 dep/.env 覆盖生成；`build.sh -s` 期间 `run-consul.sh`/`run-nomad.sh` 把 **bootstrap 出来的真 token 写回这里**。`deploy.sh` 渲染 job 用的就是它 |
| `/data/nomad/acl.token`、`/opt/consul/acl.token` | 首次 bootstrap 时持久化的真 token（chmod 600），重跑 `-s` 加入已有集群时从这里恢复 |
| `/root/.e2b/config.json` | `deploy.sh` 首次 seed-db 后写入的 **SDK 凭据**（accessToken/teamApiKey/teamId），benchmark 的 `sync-env.sh` 从这里同步 |
| `/opt/e2b-infra/nomad/*.hcl` | job **渲染模板**（含 `${VAR}` 占位） |
| `/opt/e2b-infra/rendered/*.hcl` | `deploy.sh` 用 envsubst 渲染后的**提交产物**（真值） |

记住一条铁律（详见 `single-node-offline-deploy.md` §5.0/§6.1）：
**`build.sh -i` 会把顶层 `.env` 重置成占位符 token**。集群在跑时别随手 `-i`，
否则接下来 `deploy.sh` 提交 job 会 403。

## 6. 宿主机目录地图

| 目录 | 谁创建 | 内容 |
|---|---|---|
| `/opt/e2b-infra/` | RPM | 部署根：`bin/`（全部二进制+Dockerfile+迁移文件）、`nomad/`（job 模板）、`dep/`（离线依赖+基线配置）、`build.sh`、各 start/run/install 脚本、`helm/` |
| `/opt/e2b-infra/harbor/` | `-i` 解压 | Harbor 离线安装目录（docker-compose.yml、install.sh、配置） |
| `/opt/e2b-infra/rendered/` | `deploy.sh` | 渲染后的 job 文件 |
| `/etc/nomad.d/` | `run-nomad.sh` | Nomad 配置（`default.hcl`，`-s` 末段追加 client 块） |
| `/data/nomad/` | `run-nomad.sh` | Nomad 数据（raft 状态、alloc 目录、`acl.token`） |
| `/opt/consul/` | `install/run-consul.sh` | Consul 安装+数据（`acl.token`） |
| `/orchestrator/{sandbox,template,build}` | `start-client.sh` | orchestrator 工作目录（沙箱运行数据、模板缓存、构建目录） |
| `/fc-envd/`、`/fc-kernels/`、`/fc-versions/` | `init-client.sh` | envd 二进制、客户机内核、firecracker 二进制（版本化目录） |
| `/fc-vm/` | `start/init-client.sh` | FC 运行时挂载点 |
| `/mnt/snapshot-cache`（tmpfs 65G）、`/mnt/hugepages`（hugetlbfs） | `start/init-client.sh` | 快照缓存、大页挂载 |
| `/root/data/minio/` | `-i` | MinIO 对象数据（模板、构建缓存桶） |
| `/usr/bin/orchestrator`、`/usr/bin/template-manager` | `start-client.sh` 从 `bin/` 复制 | **实际运行的**业务二进制（rpm 升级不会自动刷新这里，要手动 cp，见 runbook 场景二） |

## 7. 端口速查表

| 端口 | 组件 | 用途 |
|---|---|---|
| 80 | iptables → 3002 | 沙箱域名入口（nginx 的 80 vhost 被此规则遮蔽） |
| 443 | nginx | `https://harbor` 反代 → 127.0.0.1:2900（模板构建拉镜像用） |
| 2900 | Harbor | 镜像仓库 HTTP（docker push/pull、UI） |
| 3000 / 5009 | api | REST / gRPC |
| 3001 / 3002 / 3003 | client-proxy | edge API / 沙箱流量代理 / 健康检查 |
| 4646 / 4647 / 4648 | Nomad | HTTP+UI / RPC / serf |
| 5007 / 5008 | orchestrator | proxy / gRPC（template-manager 同端口同进程） |
| 5432 | PostgreSQL | 元数据库 |
| 6379 | redis | 沙箱运行态 |
| 8500 / 8600 | Consul | HTTP+UI / DNS |
| 9000 / 9001 | MinIO | S3 API / 控制台 |
| 53 | dnsmasq | 宿主 DNS（*.e2b.app、*.consul 转发） |

## 8. 生命周期总览：从裸机到可用

```
裸机（openEuler aarch64，自备 docker + 自编译 nbd.ko）
  │  rpm -ivh e2b-infra-2026.09-4.aarch64.rpm
  ▼
/opt/e2b-infra 就位（二进制/脚本/模板/离线依赖描述见 02、03 篇）
  │  放置离线大件到 dep/ + docker load 三个镜像（postgres/redis/debian）
  │  bash build.sh -i          ← 04 篇
  ▼
周边依赖就绪：postgres 已跑、minio 已跑、harbor 已解压待装、nginx 已跑、SDK 已装+打补丁
  │  bash build.sh -s          ← 05 篇
  ▼
harbor 起 → consul/nomad 起（ACL bootstrap）→ 宿主机调优 → 镜像构建推送 → 4 个 job 提交 → seed 用户
  │  python build_prod.py base （构建首个模板）
  ▼
SDK Sandbox.create() 可用；日常运维见 06 篇；改动迭代见 single-node-offline-deploy.md §5
```
