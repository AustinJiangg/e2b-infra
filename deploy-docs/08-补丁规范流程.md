# 08 - 补丁规范流程（双仓库协作，单一补丁模型）

> 自 2026-07 起，本项目采用**单一补丁**模型：`e2b-infra-arm` 源码仓库是唯一的
> 源码真相源，本仓库的 `0001-adapted-for-arm-architecture.patch` 由脚本从它
> 机械生成。**patch 文件永远不手改。**（§03 的 6.1 节临时工作树方法已废弃）

## 1. 两个仓库的职责

| 仓库 | 角色 | 内容 |
|------|------|------|
| `e2b-infra-arm` | **源码真相源**，所有 Go/脚本/hcl 改动都在这里做 | 官方 2026.09 基线（tag `upstream-2026.09`）+ 其上的适配提交 |
| `e2b-infra`（本仓库） | **RPM 打包物料库**，不承载源码开发 | 官方源码+vendor 的 tarball（LFS）、单一 patch、spec、e2b-deploy 部署件、benchmark、预编译二进制 |

## 2. 源码仓库结构（极简，无提交纪律要求）

```
HEAD  ← 任意数量的适配/开发提交，怎么提交都行
  ⋮
upstream-2026.09 (tag)  ← 官方 e2b-dev/infra 2026.09 原样导入，唯一锚点
```

补丁 = `git diff upstream-2026.09 HEAD` 的全量差异，与提交数量/结构无关。
不再区分多个 patch，也不需要 rebase/fixup 纪律。

## 3. 日常循环（唯一需要记住的流程）

```bash
# ① 源码仓库改代码、验证、随意提交
cd e2b-infra-arm
vim packages/orchestrator/... && go build ./... && git add -A && git commit -m "..."

# ② 生成补丁（直接写入构建仓库根目录）
cd ../e2b-infra && scripts/gen-patches.sh ../e2b-infra-arm

# ③ 构建验证（服务器上）
rpmbuild -bb e2b-infra.spec --define "_sourcedir $PWD"
rpm -Uvh --force ~/rpmbuild/RPMS/aarch64/e2b-infra-*.rpm

# ④ 通过后两个仓库分别 commit + push
```

## 4. vendor 与 tarball 规则（唯一的坑）

- LFS 里的 `e2b-infra-2026.09.tar.gz`（约 143MB）= 官方源码（约 2.8MB）+
  预生成的 `vendor/` 目录。rpmbuild 用 `-mod=vendor` **纯离线**编译。
- 日常改代码（包括加日志、改逻辑、用已有依赖库）**不需要动 tarball**。
- **仅当** go.mod 引入了新的第三方模块（或升级版本）时：在源码仓库
  `go mod tidy && go mod vendor`，把新的 vendor 打进 tarball 重新上传 LFS。
  `gen-patches.sh` 会在每次运行时提醒这条规则。

## 5. 插桩即接口：改埋点要联动 benchmark

`[ResumeSandbox] <key> cost: <数值> ms, traceID=<hex>` 是 benchmark 工具链的
解析格式（`benchmark/parse_report.py` 的 `PHASE_ROWS`、`benchmark/README.md`
阶段表）。新增/删除/改名埋点时，同步更新这两处，否则报告出现空列或漏项。
新埋点必须遵守该格式（不带额外尾缀字段，traceID 结尾）。

## 6. 升级官方 tag

1. 下载新 tag 源码，源码仓库重打基线：新的孤儿基线提交 + 移动
   `upstream-2026.09` 风格的 tag（改用新版本号命名），把适配层
   `git diff` 出来再 apply 到新基线上解冲突；
2. 重新 `go mod vendor`，按 spec 命名规则重打 tarball 上传 LFS；
3. spec 更新 `%global tag`；`gen-patches.sh` 中的 tag 名同步修改。

## 7. 禁止事项

- ❌ 手改 patch 文件（本流程下没有任何理由需要）
- ❌ 在构建仓库直接改 Go 源码逻辑（它没有源码树；发现问题回源码仓库改）
- ❌ 绕过 gen-patches.sh 用其他方式生成 patch（保证格式/头部稳定）

## 8. 配套脚本

| 脚本 | 用途 |
|------|------|
| `scripts/gen-patches.sh <arm仓库路径> [ref]` | 生成单一 patch 并写入本仓库根目录 |
| `scripts/verify-baseline.sh <arm仓库路径>` | 服务器上校验 LFS tarball 的源码部分与 `upstream-2026.09` 基线一致（忽略 vendor/），升级 tag 或怀疑 tarball 被改动时跑一次 |
