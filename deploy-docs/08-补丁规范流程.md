# 08 - 补丁规范流程（双仓库协作 SOP）

> 本文是 §03 文档 6.1 节的升级版：以 **e2b-infra-arm 源码仓库的 git 分层历史**为唯一真相源，
> 用 `git format-patch` 机械化地生成本仓库的 patch 文件。**patch 文件永远不手改。**

## 1. 两个仓库的职责

| 仓库 | 角色 | 内容 |
|------|------|------|
| `e2b-infra-arm` | **源码真相源**，所有 Go/脚本改动都在这里做 | 官方 2026.09 源码 + 补丁层 commit + 本地层 commit |
| `e2b-infra`（本仓库） | **RPM 打包物料库**，不承载源码逻辑开发 | 官方源码包 tarball（LFS）、`0001/0002 patch`、`e2b-infra.spec`、e2b-deploy 部署件、预编译二进制 |

## 2. 源码仓库的分层提交模型

`e2b-infra-arm` 的分支历史必须保持如下结构（自底向上）：

```
[本地层]   mooncake: 恢复 storage.go 的 Mooncake 存储接线     ← 不导出为 patch
[补丁层2]  orchestrator: selectable firecracker launch mode…  ← 导出为 0002-*.patch
[补丁层1]  Adapt E2B to the arm64 architecture…               ← 导出为 0001-*.patch
[基线层]   revert: 回退到官方 2026.09 基线 + 本地扩展          ← 树内容 = 官方源码 + 本地扩展文件
[历史]     Initial commit: E2B infrastructure (ARM)           ← 旧快照，仅作历史保留
```

规则：

1. **一个补丁层 commit 对应一个 patch 文件**。所以补丁数量不限——要加第三个 patch，
   就在补丁层顶部再放一个 commit，并在 spec 里加 `Patch3:` 一行（`%autosetup` 会按编号自动应用）。
   这直接回答了"规范流程是不是只能有 1 个 patch"：**不是**，多 patch 天然支持，而且应该按功能拆分。
2. **本地层 commit 的标题必须以 `mooncake:` 或 `local:` 开头**——`gen-patches.sh` 靠这个前缀
   识别"补丁层到此为止"。本地层放不进 RPM 的内容（Mooncake 集成、实验性插桩等）。
3. 补丁层必须紧贴基线层、连续排列；本地层永远在最顶部。
4. 基线层的树 = 官方源码 + 少量"补丁体系外"的本地文件（CLAUDE.md、build.sh、
   storage_mooncake.go、chunk.go 的 Mooncake 改动）。这些文件不在任何 patch 的覆盖范围内，
   放在基线层不影响 patch 生成。

## 3. 日常操作 SOP

### 3.1 修改某个已有 patch 的内容（最常见）

```bash
cd e2b-infra-arm
# ① 正常改代码、验证
vim packages/orchestrator/...
# ② 找到目标补丁层 commit（例如 0002 对应的那个）
git log --oneline | head -5
# ③ 把改动作为 fixup 提交并自动挤进对应层
git add -A
git commit --fixup=<0002层的commit号>
git rebase -i --autosquash <基线层commit号>     # 保存退出即可，本地层会自动跟着上移
# ④ 重新生成 patch 并复制到构建仓库
cd ../e2b-infra && scripts/gen-patches.sh ../e2b-infra-arm
# ⑤ 构建验证（服务器上）
rpmbuild -bb e2b-infra.spec --define "_sourcedir $PWD"
# ⑥ 通过后：spec 里 Release 加 1、补 %changelog，两个仓库各自 commit + push
```

### 3.2 新增一个 patch（新功能/新优化）

```bash
cd e2b-infra-arm
# 在本地层之下、现有补丁层之上插入新 commit：
git add -A && git commit -m "orchestrator: <新功能描述>"   # 先提交在顶部
git rebase -i <基线层commit号>    # 把新 commit 挪到 mooncake:/local: 提交之前
cd ../e2b-infra
# spec 增加一行: Patch3: 0003-<slug>.patch （文件名自定，与 gen-patches 输出重命名规则一致）
vim e2b-infra.spec
scripts/gen-patches.sh ../e2b-infra-arm
# rpmbuild 验证 → Release+1 → changelog → 双仓库提交
```

### 3.3 修改本地层（Mooncake 等，不进 RPM）

直接在 `e2b-infra-arm` 顶部提交（标题以 `mooncake:`/`local:` 开头），不需要动构建仓库。

### 3.4 升级官方 tag（如 2026.09 → 更新版本）

1. 下载新 tag 源码包，顶层目录改名为 `e2b-infra-<tag>` 重新打 tar，`git lfs` 替换本仓库的 tarball；
2. spec 更新 `%global tag`；
3. 源码仓库：以新基线重演一遍分层（新基线 commit → `git am` 各 patch，解冲突后
   `gen-patches.sh` 重新导出 → rpmbuild 验证）。

### 3.5 禁止事项

- ❌ 手改 `0001/0002 patch` 文件（§03 已强调；本流程下完全没必要）
- ❌ 在构建仓库直接改 Go 源码逻辑（它没有源码树，改不了；发现 patch 有问题回源码仓库改）
- ❌ 把 Mooncake/实验性代码混进补丁层（RPM 构建环境没有 Mooncake 依赖，会编译失败）

## 4. 配套脚本

| 脚本 | 用途 | 运行位置 |
|------|------|----------|
| `scripts/gen-patches.sh <arm仓库路径>` | 从源码仓库分层历史导出 patch，按 spec 声明的文件名放到本仓库根目录 | 任意（两仓库都已克隆） |
| `scripts/verify-baseline.sh <arm仓库路径>` | 解开真实 tarball，与源码仓库基线层逐文件对比，校验基线正确性 | **服务器**（需先 `git lfs pull`） |

## 5. 本次对齐（2026-07-22）的背景与遗留核验项

此前两仓库脱节：源码仓库停留在 6/9 初始版 0001 + 早期 netns 实验 + 独有插桩的快照，
而本仓库的 0001 经过 4 轮更新、0002 从无到有共 5 版。对齐时以**本仓库的 patch 为真相**
重建了源码仓库的分层历史；源码仓库原快照保留为历史首个 commit（`cf25391`），
被剥离的旧插桩/实验代码都可从中找回。

对齐后的验证结论（本会话内已机器校验）：

- 用重建后的分层历史运行 `gen-patches.sh`，导出的 0001/0002 与本仓库现有 patch
  的 **diff 正文逐字节一致**（0001 共 93 文件、0002 共 14 文件全部核对）；
  差异仅在无关紧要处：`index` 行（原 patch 手改后已陈旧，再生成的反而正确）、
  patch 内文件排列顺序、`@@` 行号偏移、尾部签名行。
- patch2 新增的 `launchplan` 包在容器内编译并通过其单元测试。

**遗留核验项**（请在服务器上执行一次 `scripts/verify-baseline.sh ../e2b-infra-arm`）：

1. `script_builder.go`、`script_builder_test.go`、`cfg/model.go` 不受任何 patch 覆盖，
   官方原貌是依据 0002 的上下文推断还原的，需与 tarball 比对确认；
2. `block/cache.go`、`fc/client.go`、`fc/process.go` 的前置 blob 与原 patch 记录的
   官方 blob 不一致（cache.go 约多 12 行），说明基线层在这三个文件的
   **hunk 上下文之外**还残留少量本地改动——不影响 patch 应用（正文已核对一致），
   但基线不纯，需按脚本报告修正（`git rebase -i` 编辑基线 commit，改回 tarball 内容，
   补丁层随 rebase 自动重放）。

基线核验通过后，跑一次 `scripts/gen-patches.sh` 完成最终收敛（此后 patch 的
`index` 行与 `@@` 行号将长期稳定）。本仓库的 patch 文件在核验前保持原版不动。
