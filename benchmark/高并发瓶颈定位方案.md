# 高并发下沙箱启动瓶颈定位方案

> 本文是**持续迭代**的设计文档:方案与代码会不断调整,直接在本文里改,不另起 v1/v2。
> 目标:把客户端感知的「启动一个沙箱」耗时,在**高并发**下拆成可独立读数的阶段,定位瓶颈。
> 基准版本:e2b-infra `2026.09` + `0001-adapted-for-arm-architecture.patch` + `0002-fc-launch-dedicated-helper.patch`。

---

## 1. 背景:三个口径,别混

一次 `Sandbox.create()` 的端到端耗时,跨了客户端 / API / orchestrator 三层。三个测量口径:

| 口径 | 量的是什么(代码边界) | 在哪测 |
|---|---|---|
| **客户端整体 `client_ms`** | `Sandbox.create()` 墙钟 = 路由 + 准入排队 + 取模板 + `ResumeSandbox` + 善后 | `run_benchmark.py` |
| **准入排队 `acquire wait`** | 等 `startingSandboxes` 信号量(在 `ResumeSandbox` **之前**,total 之外) | orchestrator 日志(本期新增) |
| **`ResumeSandbox` 总耗时 `total`** | `Factory.ResumeSandbox` enter→return,含恢复 VM + 启动 envd(`/init`) | orchestrator 日志 |

实测(高并发)量级参考:`client_ms ≈ 600ms`,`total ≈ 300ms`,其中 `等待fc启动 ≈ 250ms`。
`client_ms − total ≈ 300ms` 这块此前糊成一团,**首要怀疑是准入排队**(并发数 > `MAX_STARTING_INSTANCES_PER_NODE` 时排队)。

关键事实(已核对源码):
- orchestrator 的 `Create` 处理器在 `ResumeSandbox` 返回后**立即 return**(只 insert + 后台 goroutine),不再同步等 envd。所以 `total` 之外主要是**排队 + 路由**,不是 envd。
- envd 的「启动 + `/init` 初始化」是在 `total` 内的(即报告里「启动 envd」三行)。

## 2. 本期改动范围(刻意收窄,避免太乱)

1. **total 外**:加 1 个桩 `准入排队`(`acquire wait`)。
2. **total 内**:把 `等待fc启动`(`configured fc`)拆成 **`拉起FC进程`(`fc spawn`)** + **`等FC API socket`(`fc socket wait`)**。
3. **重命名 + 上色**:报告里「总耗时」→「**ResumeSandbox总耗时**」;解析 / 报告 / 可视化都按阶段展开,**每阶段一种颜色**。

**本期不做**:路由桩、handler-total 桩、envd-helper 内部桩、就绪B(proxy 轮询)桩。留待后续在本文追加。

### 为什么「等待fc启动」拆成这 2 段

`configure()`(`fc/process.go`)真实结构只有两处阻塞:

```
p.cmd.Start()            // 拉起进程：fork/exec 启动命令（只 fork 完就返回）
socket.Wait(fcSocket)    // 等 FC API socket：覆盖命名空间内的脚本/exec + FC 自身启动 + 等待机制
```

`等待fc启动` 与你的 **3 档启动模式**(`E2B_FC_LAUNCH_MODE`,见 `fc/mode.go`)的对应——这正是拆它的意义:

| 模式 | 影响 `拉起FC进程` | 影响 `等FC API socket` |
|---|---|---|
| `disabled`(默认) | fork `unshare`→bash,脚本里 ~8 次 fork/exec | 吞下整条 shell 管线 + FC 启动,**10ms 轮询** |
| `netns-exec` | 仅把末尾 `ip netns exec` 换 helper | 略减,仍 10ms 轮询 |
| `launch` | 经 `unshare --propagation unchanged` 包装 fork 极小的 fc-launch helper(mount+setns+execve 一把梭;mount ns 在子进程创建,spawn 不抢 namespace_sem) | **inotify 事件**,无轮询毛刺 |

> `cmd.Start()` 只 fork 不等脚本跑完;高并发下从庞大的 orchestrator 地址空间 fork 100 份本身也可能不便宜,拆出来才看得到。
> 想再往 `socket.Wait` 里拆(shell 管线 vs FC 自身启动)需要在 helper/脚本里打点,跨进程且分模式,本期不做。

## 4. 打点位置(代码锚点 + 日志格式)

均加进 **`0001-adapted-for-arm-architecture.patch`**(`utils.go` 为新增 section;`process.go` 在既有 section 追加 2 个 hunk)。`0002` 不动(它只改 `NewProcess`,在 `configure` 之前,互不影响)。

### 4.1 准入排队 — `server/utils.go` `waitForAcquire`
- 需补 import:`time`、`go.uber.org/zap`。
- 取 `traceID := span.SpanContext().TraceID().String()`(`wait-for-acquire` span)。
- 包住 `s.startingSandboxes.Acquire(ctx, 1)`,成功后打:
  `[ResumeSandbox] acquire wait cost: %.3f ms, traceID=%s`
- 注:`maxStartingInstancesPerNode` patch 默认 500、`acquireTimeout` 300s,均可被环境变量 `MAX_STARTING_INSTANCES_PER_NODE` 覆盖。排队大不大取决于它:并发 100 而 cap=30 → ~70 个排队;cap=500 → 几乎不排队。

### 4.2 拆 `等待fc启动` — `fc/process.go` `configure`
- 取 `traceID := childSpan.SpanContext().TraceID().String()`(`configure-fc` span,与 total 同 traceID)。
- `p.cmd.Start()` 成功后:`[ResumeSandbox] fc spawn cost: %.3f ms, traceID=%s`
- `socket.Wait(...)` 成功后:`[ResumeSandbox] fc socket wait cost: %.3f ms, traceID=%s`
- 保留 `Resume` 里原 `configured fc cost`(父);`configured fc ≈ fc spawn + fc socket wait`。

### 4.3 重命名 — 仅 `parse_report.py` 显示名
`total` 这行显示名从「总耗时」改「ResumeSandbox总耗时」(日志 key 不变)。

## 6. 可视化(`visualize_intervals.py`)— 合并甘特 + 单色 total 甘特 + 分阶段堆叠(共 3 张)

主图把三类信息**合成一张** `timeline.png`:每个沙箱一条,**按真实时刻**摆放各阶段、**按阶段上色**;
并行段在同一条内用**泳道**分层,直接看到重叠。此外仍单独输出两张更朴素的原始图——
`total_gantt.png`(单色 total 甘特)和 `stage_durations.png`(各阶段 duration 堆叠条)。
3 张图 y 轴都用从上到下 1..n 的简单编号(一行一个沙箱,按开始时间排序)。

**数据来源(关键)**:每条 `[ResumeSandbox] X cost` 日志**自带时间戳**(≈该阶段结束时刻),
区间 = `[ts − 时长, ts]`;同一沙箱的所有阶段都来自**同一 orchestrator 进程/同节点时钟**,
天然自洽、无跨机漂移。`parse_report.py` 据此写 `timeline.csv`(每行一个「沙箱-阶段」区间,
相对全局最早事件的 ms 偏移),`visualize_intervals.py` 读它画图。

**画法**:
- x 轴 = 距全局最早事件的真实时间(ms/s);y 轴 = 每沙箱一条,**按最早事件时刻排序**(≈进入顺序)。
- 每个阶段一段彩色条,摆在它的真实 `[start, end]`;**只画叶子阶段**,排除父区间(`configured fc`/
  `resume VM`/`total`)与被 `start envd` 覆盖的 envd 子段,避免重复绘制。
- **并行重叠用泳道**:同一沙箱内若区间重叠(`configure`∥`uffd`∥`rootfs`),贪心分到不同泳道,
  在该条的固定高度内分层显示——**真实并发形态**,不再折叠成关键路径。
- **未埋点的间隙=留白**(如 GetTemplate),诚实可见;无需「其他」兜底块。

配色(英文图例避免缺中文字体显示方块):准入排队=灰、`fc.NewProcess`=青、`fc spawn`=橙、
`fc socket wait`=红(通常最大)、`uffd sock wait`=蓝、`rootfs`=橄榄、`load snapshot`=紫、`start envd`=绿。

**这张图一眼能看到**:高并发下灰色「准入排队」段排成**阶梯**(每 cap 个一波);红色 `fc socket wait`
有多长、是否随波次变长(资源饱和);蓝色 `uffd` 在红色下方的泳道里**与之并行**。合并图输出
`report/timeline.png`,另两张朴素图为 `report/total_gantt.png` / `report/stage_durations.png`。

## 7. 高并发实验方法

1. **3 档模式 A/B/C**:同并发(如 100)分别 `E2B_FC_LAUNCH_MODE=disabled / netns-exec / launch`,比 `拉起FC进程` 与 `等FC API socket`。预期 netns-exec/launch 压前者,launch 的 inotify 压后者的轮询毛刺。**mode 写进 `meta.json`** 便于报告标注。
2. **并发 1 vs 100**:逐段算放大倍数,看哪段对并发最敏感。
3. **cap 扫描**:固定并发 100,扫 `MAX_STARTING_INSTANCES_PER_NODE`=15/30/100/500,看 `准入排队` 与 `等FC API socket` 此消彼长——若放开 cap 只是把时间从排队搬到等FC,说明信号量在保护已饱和的 CPU,真天花板在 FC 启动。

### 判读决策树

| 哪段大 / 随并发放大最猛 | 结论 | 方向 |
|---|---|---|
| 准入排队 | cap < 并发,卡排队 | 调 `MAX_STARTING_INSTANCES_PER_NODE`/加节点(先做 cap 扫描确认放开不拖垮 FC) |
| 拉起FC进程 | fork 启动命令贵 | 换 launch 档、posix_spawn/vfork、减 fork 次数 |
| 等FC API socket | FC 自身启动 + 轮询毛刺 | launch 档(inotify)、给 FC 更多 CPU、绑核、预热 |
| 其余(派生) | 路由/API/取模板 | 后续再补 handler/路由桩(见 §9) |
